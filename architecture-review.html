02
Frontend · 前端架构

六个真实存在的问题

前端是 v4.0 重写最彻底的部分，6 个页面 + 2 个组件 + 1 个档案管理器 + 1 个 API 客户端。表面完整，但底层有六个结构性问题。

Critical无错误边界
任何一个 API 调用抛错，整个页面白屏
App.tsx 没有 <ErrorBoundary>。当 DeepSeek API 返回 500（例如 Key 未配置），getChallenge() 抛出 Error，React 组件树崩溃，用户看到空白。当前各页面用 try-catch + setError 兜底，但只覆盖了"已预见"的错误，未预见的渲染错误依然白屏。
在 App.tsx 的 <Routes> 外层包一个 ErrorBoundary 组件，捕获渲染错误并显示降级 UI + "重新诊断"入口。
Critical数据无界增长
impactHistory 持续累积，localStorage 5MB 上限迟早爆掉
profileManager.recordImpact() 每次 push 一条完整记录（含 title、reflection 文本）。用户用 3 个月、每天 3 篇，impactHistory 约 270 条，每条约 200 字，约 54KB——看似不大。但 profile 是整体 JSON 序列化进 localStorage，加上 readHistory、milestones、currentExposure，且每次 saveProfile 都是全量覆盖。一旦数据增长或 schema 变更，JSON.parse 失败 → 档案丢失。
1) impactHistory 只保留最近 100 条 + 聚合统计；2) 添加 profileVersion 字段，支持 schema 迁移；3) saveProfile 包 try-catch，配额不足时自动归档。
Warn路由守卫分散
6 条路由各自写 hasProfile ? : <Navigate to="/scan" />，逻辑重复
App.tsx 每条 <Route> 都重复 hasProfile ? <Page /> : <Navigate to="/scan" />。新增页面必须记得加守卫，容易遗漏。且 /read/:id 没有验证 :id 是否合法。
封装 <ProtectedRoute> 组件，统一处理档案检查 + 重定向。
Warn无加载骨架
页面切换时白屏闪烁，DeepSeek 调用 3-8 秒期间无反馈
ChallengePage 调 getChallenge() 期间显示 "加载中..." 文本，但页面布局未占位，内容加载完后跳动。ReaderPage 同理。首屏 DiagScanPage 进入 chat 阶段时，教练回复前的 loading 态虽有动画，但整个对话区高度未稳定。
为每个页面定义骨架屏（Skeleton），保持布局稳定；DeepSeek 调用统一走 Suspense + lazy。
Warn状态管理职责不清
Zustand 只存 profile，但 API 调用状态（loading/error）散落各页面
useAppStore 有 loading / error 字段，但每个页面又各自维护本地 useState<boolean> 的 loading。两套 loading 状态并存，store.setError 几乎没被调用。状态来源不统一，调试困难。
要么把所有异步状态收进 store（用 createAsyncThunk 思路），要么彻底放弃 store 的 loading/error，每个页面自治。不要两套并存。
Info无数据导出
"认知数据资产"是护城河，但用户无法导出自己的档案
profileManager 没有 exportProfile() / importProfile()。用户用 3 个月积累的档案锁死在 localStorage，清缓存即丢失。"资产"无法备份，就不是真正的资产。
添加 exportProfile(): string（返回 JSON）和 importProfile(json: string)，在设置页提供导出/导入按钮。
03
Backend · 后端架构

伪算法与设计债

后端是问题最集中的地方。PRD v4.0 写得很漂亮——"三维决策引擎"、"认知冲击度预测"、"DeepSeek 评估"——但实现与文档之间存在显著差距。

PRD 说"冲击度由 DeepSeek 评估（不是简单余弦距离）"，实现却是一个 5×5 的硬编码类别距离矩阵。

Critical伪算法
"三维决策引擎"实际是 5×5 硬编码矩阵 + 加权求和
recommender.ts 的 CATEGORY_DISTANCE 是一个手写的 5×5 表（entertainment / life / knowledge / humanities / science 之间的距离）。"冲击度"= 查表取平均值。"可接受度"= 根据难度等级对距离做简单变换。"综合分"= 三个分数加权求和，权重也是硬编码。

这不是"预测"，是 if-else。任何工程师 10 分钟能复制。PRD 承诺的"DeepSeek 评估认知距离"完全没有实现——generateChallenge() 只负责生成内容文本，不参与决策。
两条路：1) 诚实路线——承认这是规则引擎，PRD 改写，去掉"AI 预测"措辞；2) 真正路线——增加 evaluateCognitiveDistance() 函数，让 DeepSeek 在决策阶段评估"盲区维度与用户高频领域的认知距离"，返回 0-1 分数。
Critical无缓存
每次 /challenge 都调用 DeepSeek，3-8 秒延迟 + 成本失控
generateDailyChallenge() → generateChallenge() → chatCompletion()，每次请求都是全新 DeepSeek 调用。用户每天点"每日挑战" = 1 次调用；点 3 张卡片各看详情 = 3 次调用；自评反哺 = 1 次调用。单用户单日 ≥5 次 DeepSeek 调用。无缓存，无去重，无批处理。
1) /challenge 结果按 userId + date 缓存 24h（前端 localStorage 即可）；2) getContentDetail 按 dimensionId + difficultyLevel 缓存；3) 批量生成：一次 generateChallenge 同时生成 3 篇，不再逐篇调详情。
WarnRESTful 违规
POST /map 和 POST /growth 是读取操作，却用 POST
/api/agent/map 接收 exposure 返回认知地图（纯计算），/api/agent/growth 接收 profile 返回成长曲线（纯聚合）。两者都是幂等读取，却用 POST。虽然 Serverless 场景下 GET + body 也不规范，但至少应该用 GET + query params。
/map?exposure=... 改 GET（exposure 太大时保留 POST 但加 Cache-Control）；/growth 同理。
Warn无请求校验
req.body 不做 schema 校验，畸形请求直接 500
每个路由直接解构 req.body，没有 zod / joi 校验。exposure 期望 Record<string, number>，但如果前端传 null，Object.entries(null) 抛错 → 500。错误信息暴露内部逻辑。
引入 zod，每个端点定义请求 schema，校验失败返回 400 + 明确字段错误。
Warn无速率限制
DeepSeek API 成本无控制，单用户可无限调用
没有 rate limiting。恶意用户可以在 1 分钟内调用 /challenge 1000 次，每次触发 DeepSeek 调用，直接打爆 API 配额。
基于 IP 的简单限流（express-rate-limit），每 IP 每分钟 10 次。生产环境加 Vercel Edge 配额。
Info无状态传输浪费
每次请求传完整 profile，但大部分字段用不到
/challenge 只需要 exposure + readHistory + difficultyLevel + highExposureFields，但 apiClient 的 getApiProfile() 把整个 profile（含 impactHistory 全量历史）都发了。3 个月后单次请求体可能 50KB+。
每个端点定义所需的 profile 子集，apiClient 按端点裁剪传输数据。
04
Data · 数据架构

资产还是负担

PRD 把"认知数据资产"列为第一护城河，但当前数据架构既不是资产，也无法成为资产——它是一个随时可能丢失的 localStorage JSON。

PRD 承诺实际实现差距
初始认知档案✅ 存于 localStorage无版本号，schema 变更即损坏
暴露历史⚠️ 只存当前值，无历史快照PRD 说"每次阅读后更新"，但没有历史版本
冲击历史✅ impactHistory 数组无上限，无聚合，无归档
成长曲线⚠️ 从 impactHistory 派生不是独立存储，每次 /growth 重新计算
认知指纹❌ 完全未实现PRD 写了，代码里没有
里程碑✅ milestones 数组触发逻辑在 assessor.ts，简单按数量
教练记忆❌ 未实现 keyInsightsprofile 有字段但从未写入
数据导出/导入❌ 未实现"资产"无法备份
Critical数据易失
localStorage 不是数据库，清缓存 / 换浏览器 / 换设备 = 档案丢失
用户用 3 个月建立的认知档案，清一次浏览器缓存就全没了。这不是"数据资产"，是"数据赌注"。PRD 说"MVP 阶段 localStorage，进阶阶段后端持久化"——但如果护城河是数据资产，这个"进阶阶段"应该是 MVP 的一部分。
1) 短期：实现 exportProfile / importProfile，用户手动备份；2) 中期：接入 Supabase 匿名账号，profile 自动云端同步；3) 长期：多设备同步 + 数据迁移。
WarnPRD 未实现项
认知指纹、教练记忆、暴露历史快照——PRD 写了，代码没有
CognitiveProfile 接口有 coachMemory.keyInsights: string[]，但 profileManager 从未写入这个字段。"认知指纹"在 PRD 2.2 节列为数据资产，但代码里完全不存在。"暴露历史"PRD 说"每次阅读后更新"，但只更新当前值，不存历史快照——GrowthPage 的成长曲线只能从 impactHistory 派生，无法展示"30 天前我的暴露值是多少"。
要么实现，要么从 PRD 删除。不要让文档和代码说两套话。
05
Model · 模型架构

Prompt 不是工程

DeepSeek 在系统中扮演"认知成长教练"，但当前的"教练方法论"完全依赖 prompt 自然语言描述，没有任何结构化约束。这意味着方法论是否生效，完全取决于 LLM 当次生成的随机性。

Critical方法论无约束
"苏格拉底式追问"只是 prompt 里的一句话，LLM 可以完全不遵守
coach.ts 的 system prompt 写了"1. 苏格拉底式追问：不直接给答案"，但没有任何机制验证 LLM 是否真的追问了。DeepSeek 完全可能直接给出百科式答案——因为没有结构化约束。当前的"方法论"是建议，不是规则。
1) 输出结构化：要求 LLM 返回 { method: "socratic" | "analogy" | "counterfactual", content: string }，前端按 method 渲染不同样式；2) 后置校验：检查回复是否以问号结尾（苏格拉底式）、是否包含类比关键词（类比桥接）；3) 失败重试：不符合方法论约束时重新生成。
Warn长期记忆=全量塞入
把整个 profile 塞进 system prompt，不是"记忆"，是"上下文"
buildCoachContext() 把高频领域、盲区、已爆破、难度、周数、累计阅读、最近自评全部拼成字符串塞进 system prompt。这不是"长期记忆"，是"每次对话都重复一遍背景"。coachMemory.keyInsights 字段存在但从未写入——真正的"记忆"（教练记住用户说过的关键洞察）没有实现。
1) 实现 keyInsights：每次对话后，让 DeepSeek 提取 1 个关键洞察存入 profile；2) 下次对话只注入最近 3 条 keyInsights，不是全量 profile；3) 控制 token 预算。
Warn无流式响应
用户等待 3-8 秒才看到完整回复，体验差
chatCompletion() 用 stream: false，等 DeepSeek 生成完整回复才返回。教练对话场景下，用户盯着"教练正在思考..."8 秒，体验不佳。DeepSeek 支持 SSE 流式，但未使用。
/coach 和 /diagnose 改为 SSE 流式，前端逐字渲染（DiagScanPage 已有 Typewriter 组件可复用）。
Info上下文截断
diagnoseConversation 只传最后 10 条消息，早期对话丢失
llm.ts 第 89 行 messages.slice(-10)。如果诊断对话超过 10 轮（用户话多），最早的用户回答会被截断，导致最终 analyze() 拿到的汇总输入不完整。虽然当前固定 4 轮，但硬编码截断是隐患。
不按条数截断，按 token 数截断（tiktoken 估算），优先保留首轮和最近轮。
06
Moat · 护城河真相

四层逐一拆解

PRD v4.0 宣称四层护城河。现在诚实地评估：每一层的"护城河"成色如何？是真正的壁垒，还是营销话术？

护城河PRD 声称实际成色可复制性
① 认知数据资产 用户用得越久，档案越完整，迁移成本越高 弱 — localStorage 易失，无导出，无云端，清缓存即归零 任何竞品都能做 localStorage，无壁垒
② 反推荐算法 三维决策引擎，认知冲击度预测 弱 — 5×5 硬编码矩阵 + 加权求和，10 分钟可复制 完全可复制，无技术壁垒
③ 交互深度 诊断扫描 + 仪式挑战 + 闭环反馈 + 成长可视化 中 — 流程完整，但任何团队 2 周可复刻 体验层护城河最浅，靠执行速度
④ 教练角色 有方法论的专家，不是通用助手 中 — prompt 可复制，但"品牌认知"需要时间建立 prompt 1 秒复制，品牌无法复制
当前的四层护城河，三层是"功能描述"不是"壁垒"。真正的护城河必须是"竞品知道了也做不到"的事——不是 prompt，不是矩阵，是数据网络效应或用户迁移成本。
Path真正的护城河方向
从"功能护城河"转向"数据护城河"
当前架构是单机的，用户数据互不相通。真正的护城河在于：群体认知数据。当系统知道"1000 个娱乐高频用户中，68% 在接触叙事学后冲击自评 ≥4 星"，这个群体统计模型是任何新竞品无法获得的——他们没有用户数据。这才是"竞品知道了也做不到"的事。
1) 后端持久化用户数据（匿名）；2) 聚合"维度 × 高频领域 × 冲击分"统计模型；3) 用群体数据反哺个人推荐（"和你类似的人，从天文学获得了 5 星冲击"）。这才是数据护城河。
07
Refactor · 重构方向

从伪到真的路径

不是推倒重来，是分层做实。当前 v4.0 的骨架是对的（7 阶段 Pipeline、6 页面、教练角色），但每一层都需要从"看起来像"变成"真的是"。

重构优先级（P0 → P2）
P0 — 必须做（否则产品不可用）
 ├── 删除 5 个死代码文件
 ├── 添加 ErrorBoundary
 ├── 添加 profileVersion + schema 迁移
 ├── impactHistory 上限 100 条 + 聚合
 └── DeepSeek 调用添加缓存（前端 localStorage）

P1 — 应该做（否则护城河是假的）
 ├── 实现认知指纹（profile 派生 + 可分享）
 ├── 实现数据导出/导入
 ├── 教练方法论结构化（输出 method 字段 + 后置校验）
 ├── 真正的 DeepSeek 评估认知距离（替换 5×5 矩阵）
 └── SSE 流式响应（/coach + /diagnose）

P2 — 可以做（长期价值）
 ├── 后端持久化（Supabase 匿名账号）
 ├── 群体认知统计模型（数据网络效应）
 ├── keyInsights 教练记忆
 ├── 暴露历史快照（独立存储，非派生）
 └── 请求校验（zod） + 速率限制
Decision架构决策点
三个需要先回答的问题，再写代码
1. 算法层：承认"三维决策引擎"是规则引擎（PRD 改写），还是投入做成真正的 DeepSeek 评估？
2. 数据层：MVP 阶段是否引入后端持久化？如果护城河是数据资产，localStorage 不够。
3. 护城河层：是否转向"群体数据网络效应"路线？这需要后端 + 用户量，不是前端能解决的。
